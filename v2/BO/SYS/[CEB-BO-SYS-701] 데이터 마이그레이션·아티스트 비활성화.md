## 2. 화면 목적

운영 중인 아티스트가 다음과 같은 라이프사이클 이벤트를 겪을 때, **콘텐츠·팬덤·로그·자산 데이터를 어떻게 처리할지** 표준 정책을 정의한다.

| 이벤트 | 의미 | 가역성 |
| --- | --- | --- |
| **비활성화** (Inactive) | 운영 중단. 신규 콘텐츠 차단, 기존 read-only | 가역 (재활성화 가능) |
| **통합** (Merge) | 두 그룹을 하나로 합병 (예: 솔로 → 그룹 합류) | 비가역 |
| **삭제** (Delete) | 영구 종료. 데이터 보관 기간 경과 후 영구 삭제 | 일정 기간 가역 (60일 grace) |

> 본 화면은 **정책 정의서**다. 실제 운영 콘솔(UI)은 Phase 3에서 [CEB-BO-SYS-701-CONSOLE] 등으로 별도 작성.

## 3. 비활성화 정책

### 3.1 트리거

- ART 영역 그룹 상세에서 SUPER가 [비활성화] 클릭 → 확인 모달 → 처리.

### 3.2 즉시 처리

| 영역 | 처리 |
| --- | --- |
| **앱 노출** | 홈 화면 아티스트 선택 dropdown에서 즉시 제외 (회원이 선택 중이면 NULL로 fallback) |
| **콘텐츠 신규 생성** | Quest·BIVE·티켓·서포트 신규 생성 차단 (생성 화면에서 해당 아티스트 옵션 숨김) |
| **기존 콘텐츠** | read-only. 진행 중인 Quest는 종료 옵션 또는 자동 종료 (관리자 결정) |
| **팬덤 레벨·덕력 누적** | 회원 누적은 보존. 신규 적립 차단 |
| **응모권 발급** | 해당 아티스트 출처 발급 차단. 보유 잔액은 그대로 |
| **EXTERNAL_ARTIST 권한** | 자동 만료 처리 (만료일 즉시 도래) + 운영자 알림 |
| **활동 로그** | 그대로 보존 (3년) |

### 3.3 가역성

- 비활성화 후 SUPER가 [재활성화] 클릭 시 즉시 복원.
- 재활성화 시 EXTERNAL_ARTIST 권한은 자동 복원되지 않음 (수동 재매핑).

## 4. 통합 정책 (Phase 3)

### 4.1 시나리오

- 솔로 아티스트 A가 그룹 G에 합류 → A의 모든 데이터를 G로 이전.
- 두 그룹 X·Y 합병 → X·Y 데이터를 새 그룹 Z로 이전 또는 X·Y 중 하나로 흡수.

### 4.2 처리 방안 — 레퍼런스 갱신 (Reference Update)

| 데이터 | 처리 |
| --- | --- |
| **Quest·Raffle** | `artistId` 필드 갱신 (A → G). 기존 검수 이력은 변경 없음 |
| **BIVE 토큰** | `artistGroupId` 갱신. 토큰 이름·메타데이터는 그대로 (NFT 일관성) |
| **팬덤 레벨** | A 누적 EVT를 G 누적 EVT에 합산. 회원별 합산 로직 [CEB-BO-EVT-501] 참조. 합산 시 곡선 재계산 비동기 처리 |
| **덕력 시즌** | A 시즌과 G 시즌이 동일하면 합산. 다르면 정책 결정 (운영자 입력) |
| **응모권 발급 출처** | `sourceArtistContext` 갱신 (A → G) |
| **활동 로그** | 로그 자체는 변경 안 함. 단 신규 로그는 G로 기록 |
| **반려사유 마스터** | A·G 양쪽 사유 머지 (중복 코드 시 G 우선) |
| **곡선·시즌·BIVE 등급 매핑** | G의 정책 유지. A의 정책은 아카이브 |

### 4.3 통합 작업 단계

1. 시뮬레이션 — 통합 결과 미리보기 (영향 데이터 건수, 충돌 항목)
2. 운영자 검토 — 충돌 항목 수동 결정
3. 통합 실행 — 트랜잭션 + 롤백 가능 윈도우 (24시간)
4. 24시간 경과 — 영구 확정

## 5. 삭제 정책

### 5.1 트리거

- 비활성화 상태 그룹에서만 [삭제] 가능. 활성 그룹 직접 삭제 차단.

### 5.2 60일 Grace Period

```text
[삭제 클릭] ──→ [Soft Delete 상태] ──60일──→ [영구 삭제]
                      ↑
                      │
              [복구] 가능 (60일 내)
```

### 5.3 Soft Delete 상태 동안 처리

| 영역 | 처리 |
| --- | --- |
| 앱 노출 | 완전 차단 |
| 콘텐츠 | 신규 생성 차단. 기존도 회원에게 노출 안 함. 운영자 화면에서는 회색·비활성으로 노출 (감사용) |
| 회원 데이터 (팬덤 레벨·덕력 누적) | 보존. 회원 화면에서는 "(종료된 아티스트)" 라벨 |
| 활동 로그 | 그대로 보존 (3년) |
| EXTERNAL_ARTIST 권한 | 즉시 만료 |

### 5.4 60일 경과 후 영구 삭제

| 영역 | 처리 |
| --- | --- |
| 콘텐츠 마스터 (Quest·BIVE·티켓 등) | 비식별화 후 삭제 (메타만 보관) |
| 회원 팬덤 레벨·덕력 누적 | **영구 보관** (회원 데이터는 영구 누적 정책 [CEB-000]) |
| 회원 응모권 잔액 | 보존 (다른 아티스트 래플에 사용 가능) |
| 활동 로그 | [CEB-BO-000] §6 표준대로 3년까지 보관 |

> **회원 데이터는 영구 보관 원칙** ([CEB-000] 통합 자산 정책).

## 6. 회원 영향 처리

### 6.1 비활성화·삭제 시 회원 알림

- 회원이 해당 아티스트를 팔로우 중이면 앱 인벤토리에서 자동 언팔로우.
- 회원에게 푸시 알림: "팔로우 중인 [아티스트명]의 운영이 종료되었습니다. 누적 활동 데이터는 보존됩니다."
- 진행 중인 Quest 응모는 환불 (응모권 반환).

### 6.2 회원 잔여 자산 처리

| 자산 | 처리 |
| --- | --- |
| 응모권 (해당 아티스트 출처) | 보존. 다른 아티스트 래플에 사용 가능 |
| BIVE 토큰 (해당 아티스트) | 보존. 회원 지갑에 그대로 존재. 다만 일부 혜택은 비활성화될 수 있음 |
| 팬덤 레벨 | 보존. 회원 화면에 "(종료된 아티스트)" 표기 |
| 덕력 누적 | 보존 (전 아티스트 통합 자산) |

## 8. 활동 로그

| 액션 | 메시지 템플릿 |
| --- | --- |
| 비활성화 | 아티스트 '{group}'을(를) 비활성화했습니다. (사유: {reason}) |
| 재활성화 | 아티스트 '{group}'을(를) 재활성화했습니다. |
| 통합 시작 | 아티스트 '{from}'을(를) '{to}'(으)로 통합 작업을 시작했습니다. (시뮬레이션 결과: 영향 N건) |
| 통합 확정 | 아티스트 '{from}' → '{to}' 통합이 확정되었습니다. |
| Soft Delete | 아티스트 '{group}'을(를) 삭제 (Soft) 처리했습니다. (60일 후 영구 삭제) |
| 복구 | 아티스트 '{group}'을(를) 복구했습니다. (Soft Delete 후 {N}일 경과) |
| 영구 삭제 (자동) | 아티스트 '{group}' 60일 경과 — 영구 삭제 완료. |

| 단계 | 적용 |
| --- | --- |

## 10. 에러·예외 상황 (정책 명시)

| 상황 | 정책 |
| --- | --- |
| 활성 그룹 직접 삭제 시도 | 차단 — "비활성화 후 삭제할 수 있습니다." |
| 통합 시 회원 누적 EVT 합산 충돌 | 두 곡선이 다르면 운영자가 통합 시뮬레이션에서 충돌 항목 수동 결정 |
| 통합 시 BIVE 토큰 ID 충돌 | 차단 — 동일 ID는 통합 불가. 운영자가 토큰 ID 매핑 수동 처리 |
| Soft Delete 후 60일 경과 직전 복구 | 복구 가능. 복구 시 모든 데이터 즉시 복원 |
| Soft Delete 60일 경과 후 복구 시도 | 차단 — "복구 기간이 만료되었습니다." |
| 관리자가 자기 매핑된 EXTERNAL_ARTIST 그룹 비활성화 시도 | 차단 — "본인이 권한자로 매핑된 그룹은 비활성화할 수 없습니다. (다른 SUPER에게 요청)" |
| 영구 누적 회원 데이터를 정말로 삭제해야 하는 경우 | 본 정책 범위 밖 — 별도 GDPR 삭제 요청 처리 절차 ([CEB-BO-000] §14 회원 제재·삭제 정책 참조) |

> 본 문서는 회원 앱 운영 영역(APP)의 전체 범위·정책·서브 영역을 정의한다. 사이드바 "앱" 메뉴 하위의 모든 서브 화면 SSOT.

## 1. 영역 개요

회원 앱 운영(APP)은 운영자가 회원 앱의 노출 콘텐츠와 알림을 관리하는 통합 영역. 현재 2개 서브 영역으로 구성된다.

| 항목 | 값 |
|---|---|
| 영역 범위 | 배너 관리(슬롯 인벤토리) + 알림 관리(인앱 + 푸시) |
| 사이드바 위치 | 운영 사이트 "앱" 메뉴 하위 |
| 서브 영역 수 | 2종 (배너 / 알림) — 추후 확장 가능 |
| 영역 ID 체계 | `CEB-BO-APP-{번호}` — 배너 1xx·2xx, 알림 3xx |

### 1-1. 서브 영역

| 서브 영역 | 사이드바 메뉴 | 핵심 화면 | 용도 |
|---|---|---|---|
| 배너 | "앱 > 배너 관리" | APP-101 / APP-202 / APP-201 | 슬롯(위치 × 아티스트) 자동 생성 + 슬롯별 배너 라이프사이클 운영 |
| 알림 | "앱 > 알림 관리" | APP-301 / APP-302 | 기본알림(인앱 알림센터) + 푸시알림(Web Push) 통합 발송·예약. 재사용 템플릿 개념 자체 폐지 (APP-303 v3.1) — 운영자는 매번 직접 작성 |

# 배너 서브 영역 정책

## 4. 슬롯 정의 (배너)

### 4-1. 슬롯 핵심 개념

| 용어 | 정의 |
|---|---|
| 슬롯(slot) | 위치 × 아티스트 조합. 시스템이 자동 생성하며 운영자가 추가·삭제 불가 |
| 위치(slotKind) | 회원 앱에서 배너가 실제 노출되는 자리 (메인 / 오늘의 할일 / 다함께 / 미션) |
| 아티스트 | 전역(모든 회원, 메인 위치 전용) 또는 단일 아티스트 |
| 배너 | 슬롯 안에 등록되는 노출 콘텐츠 1건 |

### 4-2. 위치 4종 정의

| 탭 | 위치 | 타겟 모드 | 슬롯당 배너 정책 | 회원 앱 노출 형태 |
|---|---|---|---|---|
| 홈 | 메인 | 전역만 | 캐러셀, 동시 최대 8개 (운영자 정렬) | 홈 최상단 슬라이드 |
| 홈 | 오늘의 할일 | 아티스트만 | 단일 1개 (자동 교체) | 홈 미션 영역 단일 카드 |
| 아티스트 메인 | 다함께 | 아티스트만 | 캐러셀, 동시 최대 8개 (운영자 정렬) | 아티스트 홈 캐러셀 |
| 아티스트 메인 | 미션 | 아티스트만 | 단일 1개 (자동 교체) | 아티스트 홈 미션 영역 |

### 4-3. 자동 생성 슬롯 수 (활성 아티스트 5명 기준)

| 탭 | 슬롯 = 위치 × 대상 | 총 |
|---|---|---|
| 홈 | 메인 **1**(전역) + 오늘의 할일 **5**(아티스트) | **6** |
| 아티스트 메인 | 다함께 **5** + 미션 **5** | **10** |

## 5. 배너 노출 기간 + 제어 정책

### 5-1. 노출 기간 모드

| 모드 | 동작 |
|---|---|
| 무기한 | 시작·종료일 없음. 운영자가 [즉시 노출 시작]을 눌러야 노출. [즉시 노출 종료]로만 종료 |
| 사용자 지정 | 공개일·종료일 입력. 일정 도래 시 자동 노출/종료. 운영자가 [즉시 시작/종료]로 오버라이드 가능 |

### 5-2. 시점별 동작 (사용자 지정 모드)

| 시점 | 동작 |
|---|---|
| 공개일 도래 | 자동 노출 시작 (상태: 임시저장 → 노출중) |
| 종료일 도래 | 자동 노출 종료 (상태: 노출중 → 노출 종료, 데이터 보존) |
| [즉시 노출 시작] | 공개일 미도래여도 즉시 노출. 공개일 자동 갱신 |
| [즉시 노출 종료] | 종료일 미도래여도 즉시 종료. 종료일 자동 갱신 |

### 5-3. 캐러셀 슬롯 동시 노출 한도 (메인 / 다함께)

- 슬롯 1개당 동시 노출 최대 8개. 등록 자체는 무제한
- 9번째를 [노출 시작]하려고 하면 차단 + 안내 토스트
- 운영자가 직접 1개를 OFF한 뒤 9번째를 ON 해야 함

### 5-4. 단일 슬롯 자동 교체 (오늘의 할일 / 미션)

- 슬롯당 노출중(ACTIVE) 배너는 항상 1개만. 임시저장(DRAFT)은 여러 건 등록 가능
- 운영자가 대기 중인 DRAFT 배너를 [노출 시작]하면 그 슬롯의 기존 노출 배너가 자동 OFF
- 슬롯이 비어 있으면 회원 앱에서 해당 자리가 숨겨짐 (기본 폴백 이미지 없음)

## 9. 딥링크 공통 정책 (배너·알림 공유)

배너(APP-201)·알림(APP-302)이 동일한 딥링크 입력 체계를 사용한다 — 운영자가 URL을 직접 텍스트로 입력하는 단순 단일 텍스트박스 방식.

### 9-1. 입력 방식

**URL 단일 텍스트 입력**. 사전 정의된 소스 타입 분류(드롭다운)는 두지 않으며, 운영자가 회원 앱에서 도달하길 원하는 URL(내부 또는 외부)을 직접 텍스트박스에 입력한다.

- **내부 URL** (예: `/raffle/123`, `/quest/45`, `/artists/v01d`): 회원 앱의 SPA 라우팅으로 처리. 같은 탭에서 이동
- **외부 URL** (예: `https://example.com/event`, `http://…`): 새 탭에서 열림
- **빈 값**: 이동 없음 — 클릭 비활성 (배너는 표시 전용, 알림은 본문 클릭 비활성)

### 9-2. 딥링크 입력 UI 공통 규격

| 항목 | 규격 |
|---|---|
| 입력 컴포넌트 | 단순 텍스트박스 1개 (`<input type="text" />`) |
| 기본값 | 빈 값 |
| placeholder | `"https:// 또는 /로 시작하는 URL (비우면 이동 없음)"` |
| 형식 검증 | `https://` · `http://` · `/` 중 하나로 시작해야 함. 그 외 형식은 저장 시 인라인 에러 |
| 빈 값 허용 | ✅ (빈 값 = 이동 없음) |
| 외부 URL 승인 | 자유 (별도 승인 없음) — 운영자가 입력한 URL 책임 |

### 9-3. 양 영역에서의 사용 차이

| 영역 | 필수/선택 | 빈 값 의미 |
|---|---|---|
| 배너 (APP-201) | 선택 | 표시 전용 배너 (클릭 비활성) |
| 알림 (APP-302) | 선택 | 본문 클릭 비활성, 알림센터·푸시는 정상 표시 |

> **분석 추적 정책 (별도)**: 본 정책 변경으로 소스 타입 기반 채널 분석(이전 8종 enum)은 더 이상 지원되지 않는다. 분석은 URL 패턴 기반(`/raffle/`·`/quest/` 등)으로 별도 운영팀 정책에서 정의한다.

# 알림 서브 영역 정책

## 11. 알림 채널 정의

회원 앱은 **PWA(Progressive Web App)**로 운영되며, 푸시알림은 **Web Push API** 기반으로 발송된다.

| 채널 | 정의 | 회원 앱 노출 형태 |
|---|---|---|
| 기본알림 | 인앱 알림센터 (회원 앱 안의 알림 아이콘·리스트) | 회원 앱 알림 화면, 영구 보관, 회원이 읽음/삭제 처리 가능 |
| 푸시알림 | Web Push (PWA · VAPID 기반) | 회원 디바이스의 알림 배너·잠금화면. 회원 브라우저 알림 권한(`Notification.permission`), 푸시 구독(`PushSubscription`) 등록, iOS의 경우 PWA 홈 화면 추가 여부 영향 |

한 메시지는 **기본 + 푸시 동시 발송**을 기본으로 한다. 운영자는 폼에서 **푸시 발송 토글**로 푸시만 끌 수 있다 (기본알림은 항상 생성).

### 11-1. PWA 푸시 발송 제약

회원 앱이 PWA이기 때문에 네이티브 앱과 다음 점이 다르다. 운영자는 발송 규모·도달 예측 시 이 제약을 인지해야 한다.

| 환경 | 푸시 수신 조건 |
|---|---|
| Android (Chrome/Edge 등) | 브라우저 탭만 있어도 수신 가능. 알림 권한 허용 + 푸시 구독 등록 시 |
| iOS Safari | **PWA가 홈 화면에 추가되어 있어야만** 푸시 수신 가능 (iOS 16.4+). Safari 탭만 사용하는 회원은 푸시 미수신 |
| Desktop (Chrome/Edge/Firefox) | 알림 권한 허용 + 푸시 구독 등록 시 수신 가능 |
| 브라우저 푸시 쿼터 | 일부 브라우저는 자체 발송 빈도 제한(예: Chrome budget). 우리 일일 한도(§17) 미만에서도 잘릴 수 있음 |

기본알림(인앱 알림센터)은 위 제약과 무관하게 항상 생성된다.

## 12. 알림 발송 방식

| 방식 | 동작 |
|---|---|
| 운영자 수동 | 운영자가 [CEB-BO-APP-302]에서 메시지를 직접 작성·예약·발송. 즉시 또는 예약 시점 발송. 모든 메시지·대상·발송 시점을 운영자가 자유롭게 정의 |
| 시스템 자동 트리거 | 시스템 이벤트(래플 당첨·퀘스트 제출 승인/반려·BIVE 구매 완료·회원 가입 환영 등)에 백엔드가 자동 발송. 메시지·대상·발송 조건은 모두 백엔드 코드/설정에 정의되어 있어 **백오피스에서 신규 등록·수정 불가**. 변경은 개발팀 요청 채널을 통해서만 가능. 일정 기반 캠페인(BIVE 컬렉션 출시 D-1·응모권 만료 임박 등)은 자동 트리거가 아니라 수동 알림으로 운영자가 직접 작성·예약한다 |

### 12-1. 자동 트리거 변경 절차

운영자가 자동 트리거의 메시지·대상·활성 상태를 변경하고 싶을 경우:

1. 백오피스 [CEB-BO-APP-301] 자동 트리거 탭에서 현재 정책 목록 확인 (조회 전용)
2. 변경 요청을 개발팀에 전달 (요청 채널은 별도 합의)
3. 개발팀이 백엔드 코드/설정 변경 후 배포
4. 배포 후 백오피스에서 갱신된 정책 카탈로그 확인

수동 + 자동은 [CEB-BO-APP-301]에서 **별도 탭으로 분리** 조회된다 (수동 알림 인벤토리 / 자동 트리거 정책 카탈로그).

| 대상 | 정의 |
|---|---|
| 전역 | 모든 회원 (글로벌 공지·점검 안내 등) |
| 아티스트 팬덤 | 특정 아티스트 그룹 팬덤(예: V01D 팬). 회원 팬덤 그룹 매핑 기준 ([CEB-BO-USR-201]) |
| 특정 회원 그룹 | 운영자가 회원 목록을 직접 업로드·검색·선택한 임시 그룹 (CS 응대·1:1 안내) |

> 트리거 알림(자동)은 대상이 트리거 이벤트의 주체로 자동 결정 (예: 래플 당첨자 N명).

## 14. 알림 상태 정의

수동 알림(운영자 캠페인)과 자동 트리거 알림(시스템 이벤트)은 라이프사이클이 달라 상태 enum도 별도 정의한다. [CEB-BO-APP-301]은 발송 방식별 탭으로 분리해 각 탭에서 해당 상태만 노출한다.

### 14-1. 수동 알림 상태 (6종)

| 상태 (운영 표시) | 데이터 키 | 의미 | Badge 색상 |
|---|---|---|---|
| 임시저장 | DRAFT | 작성 완료, 발송 전 | 회색 |
| 예약 | SCHEDULED | 발송 시점 예약됨 | 인디고 |
| 발송중 | SENDING | 대량 발송 처리 중 | 노랑 |
| 발송 완료 | SENT | 발송 처리 완료 (도달 수 집계됨) | 초록 |
| 부분 실패 | PARTIAL_FAILED | 일부 회원 발송 실패 (재시도 권장) | 주황 |
| 실패 | FAILED | 발송 처리 자체 실패 | 빨강 |

### 14-2. 자동 트리거 정책 상태 (운영자 노출 2종)

운영자가 [CEB-BO-APP-301] 자동 트리거 탭에서 확인하는 상태는 **트리거 정책 단위의 활성 여부 2종**만이다.

| 상태 (운영 표시) | 데이터 키 | 의미 | Badge 색상 |
|---|---|---|---|
| 사용중 | active=true | 현재 트리거가 활성화되어 이벤트 발생 시 자동 발송됨 | 초록 |
| 미사용 | active=false | 트리거 정책은 등록되어 있으나 비활성화 상태 (개발팀에 요청해 켤 수 있음) | 회색 |

> **발송 인스턴스별 상태**(SENDING/SENT/PARTIAL_FAILED/FAILED)는 백엔드 내부 상태로 백오피스에서 노출하지 않는다. 운영자에게는 트리거 정책의 활성 여부만 보이며, 개별 발송 인스턴스의 성공/실패는 관심 영역이 아니다. 활성 전환·메시지 수정은 모두 개발팀 요청 채널을 통해서만 가능하다 (§12-1).

## 17. 알림 발송 한도·정책

### 17-1. 한도·동시 발송

| 항목 | 정책 |
|---|---|
| 회원당 일일 푸시 수신 한도 | **20건** (시스템 트리거 + 수동 합산). 초과 시 푸시 미발송 (기본알림은 정상 생성) |
| 전역 알림 동시 발송 한도 | 시스템 큐 처리 — 운영자 [발송] 클릭 후 평균 5~15분 소요 |
| 회원 탈퇴·휴면 | 알림 발송 대상에서 자동 제외 |
| 운영 차단 회원 | 알림 발송 대상에서 자동 제외 |

### 17-2. 푸시 미발송 사유 정의 (PWA Web Push 기준)

운영자가 [발송]을 눌러도 다음 사유에 해당하는 회원은 푸시를 수신하지 못한다. **기본알림(인앱 알림센터)은 모든 사유에 무관하게 정상 생성**되며, 회원은 앱 진입 시 알림센터에서 확인 가능하다.

| 사유 | 정의 | 발송 결과 카운트 분류 |
|---|---|---|
| 알림 권한 미허용 | 회원 브라우저의 `Notification.permission` 값이 `granted`가 아님 (`default` 또는 `denied`) | **권한 OFF** |
| 푸시 구독 없음 | 회원이 푸시 구독(`PushSubscription`) 자체를 등록한 적 없음 (앱 첫 진입 후 권한 미응답 등) | **권한 OFF** |
| iOS PWA 미설치 | iOS Safari 회원이 홈 화면에 PWA를 추가하지 않은 상태 (iOS는 PWA 설치 회원만 푸시 수신 가능) | **권한 OFF** |
| 구독 만료·철회 | 회원 구독이 푸시 서비스에서 만료(`expired`) 또는 철회(`unregistered`) 처리됨 (브라우저 데이터 삭제 등) | **권한 OFF** |
| 일일 한도 초과 | §17-1의 회원당 일일 20건 초과 시 발송 차단 | **한도 초과** |
| 푸시 서비스 일시 오류 | Web Push 게이트웨이(VAPID 기반 push service — Chrome/Edge는 FCM 백엔드, Safari는 Apple Push, Firefox는 자체 등 브라우저별 push service) 응답 오류로 발송 실패 | **실패** (재시도 가능) |

발송 결과 통계는 위 분류를 합산해 **도달 수 / 권한 OFF / 한도 초과 / 실패**로 운영자에게 표시한다.

### 17-3. 구독 회복 정책

회원이 푸시 권한을 차단했다가 추후 다시 허용한 경우, 회원 앱이 자동으로 새 구독을 등록한다. 운영자가 별도로 재구독 요청할 필요는 없다.

## 0. 감사 개요

팬덤레벨 메뉴에 대해 **앱이 요구하는 관리 항목을 BO가 올바르게 기획하고 프로토타입에 구현했는지** 교차 검증했다.

| 감사 축 | 대상 |
| --- | --- |
| 앱 요구 | [CEB-EVT-201] 팬덤 레벨 + 앱 프로토타입 (`prototype-app/src/app/fandom/page.tsx`) |
| BO 기획 | [CEB-BO-EVT-000/101/201/401] (2026-05-29 운영 정합 개정본) |
| BO 프로토타입 | `prototype/src/app/artists/fandom/` (리스트·생성모달·상세 3탭·mock) |
| 데이터 모델 | `Celebus_고도화_덕력_팬덤_기억저장소.sql` (팬덤 레벨 8개 테이블) |

### 종합 평가

| 항목 | 점수 | 결론 |
| --- | --- | --- |
| 기획(명세) 품질 | ⭐⭐⭐⭐☆ | 우수 — 4개 문서 내부 일관성·데이터 모델 95% 정합 |
| BO 프로토타입 정합 | ⭐⭐☆☆☆ | **약 45% — 개정 이전(v1.0) 기준 제작, 전면 개정 필요** |
| 앱 ↔ BO 커버리지 | ⭐⭐⭐⭐☆ | 약 80% — 핵심 관리 항목 충족, 일부 정책 텍스트 관리 공백 |
| 데이터 모델 | ⭐⭐⭐⭐☆ | 양호 — 구조 적절, 일부 결함·미정의 |

**핵심 결론**: 기획(명세)은 운영 도입 가능 수준으로 잘 정비됨. 그러나 **BO 프로토타입이 명세 개정 이전 버전에 머물러 있어 명세와 약 55% 괴리**가 가장 시급한 후속 과제다.

## 1. 기획(명세) 품질 — ✅ 우수

- EVT-000/101/201/401 4개 문서의 시즌 상태(준비/진행중/정산중/종료)·보상 타입 5종·보상 지급 상태 6종이 데이터 모델 enum과 정확히 일치한다.
- 생성 → 곡선 설정 → 보상 설정 → 게시 → 레벨업·지급 → 추첨 → 정산 → 종료의 운영 전 과정이 BO 화면에 매핑된다.
- 본 감사로 발견된 명세 갭(§5)은 모두 정정 완료.

## 3. 앱 ↔ BO 커버리지 — 약 80%

### 정상 커버 (BO에서 완전 관리)

레벨 수·레벨별 목표 덕력·보상 타입 5종·다국어 보상명·지급 방식·당첨 인원·래플 추첨·보상/레벨업 현황·시즌 상태 전환 — 모두 [CEB-BO-EVT-101/201/401]에서 관리됨.

### 관리 공백 (앱 정책 하드코딩 — BO 변경 경로 없음)

| 항목 | 현 상태 | 검토 |
| --- | --- | --- |
| 0DUK 유저 안내 배너 문구(3언어) | 앱 정책 고정 | 아티스트·언어별 커스터마이징 필요성 검토 |
| 활동 유도 CTA 라벨·보상값 | 앱 고정 | BO 관리 필요성 검토 |
| 레벨업 축하 모달 텍스트 | 앱 고정 | 커스터마이징 선택 사항 |

→ 운영 커스터마이징 필요 여부는 제품 결정 사항 (§6 후속 권고 3).
